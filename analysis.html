Parallel Analysis: Setting an Alarm on a Smartphone

Task: Set an alarm for 6:00 AM

Norman’s Stage Real-World Example (Alarm App) GOMS Interpretation & Rationale
(GOMS 1983 vs Norman 1988)		 Alignment Analysis
Goal ↔ GOAL
(User intention)		 “I need to wake up at 6:00 AM tomorrow.” Goal: Set alarm for 6:00 AM.
Top-level objective in GOMS.		 Perfect Match
Both models begin with user intention.
Plan ↔ METHOD & Selection Rules
Mental state before action		 “I’ll open the Clock app and add a new alarm.” Method:
Open Clock → Alarm → Add Alarm

Selection rules remain implicit.		 Conceptual Match
Norman’s planning aligns with GOMS methods.
Specify Choose hour = 6, minutes = 00, AM Not explicitly modeled in GOMS Key Divergence
Cognitive specification is ignored by GOMS.
Execute ↔ OPERATOR Scroll time values and tap “Save” Operators: Scroll, Tap
KLM may include [M][P][K]		 Direct Match
Physical execution is modeled by both.
Perceive / Interpret / Evaluate Perceive: Alarm shows “6:00 AM – ON”
Interpret: Alarm is active
Evaluate: Goal achieved 		Not modeled in GOMS Fundamental Divergence
Feedback and evaluation lie outside GOMS’s scope.
Concluding Comparison:
While GOMS efficiently models goals and execution for routine, well-learned tasks, it oversimplifies user interaction by ignoring specification and feedback. In contrast, Norman’s Action Cycle offers a richer explanatory framework by explicitly addressing perception, interpretation, and evaluation, making it more suitable for analyzing real-world usability and design quality.
